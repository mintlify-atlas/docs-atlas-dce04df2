The crafting system in Aeven allows players to create items from raw materials using various crafting stations. The system supports multiple crafting skills, timed crafting mechanics, batch crafting, and recipe discovery.

## System Architecture

### Core Components

The crafting system consists of four main modules located in `src/crafting/`:

<CodeGroup>
```rust mod.rs
//! Crafting System
//!
//! Provides recipe definitions and a registry for crafting items.

pub mod definition;
pub mod registry;
pub mod stations;

pub use definition::{CraftResult, Ingredient, RecipeCategory, RecipeDefinition};
pub use registry::CraftingRegistry;
pub use stations::StationRegistry;
```
</CodeGroup>

**Module Breakdown:**
- `definition.rs` - Recipe data structures and TOML deserialization
- `registry.rs` - Recipe loading and management
- `stations.rs` - Crafting station definitions
- `crafting.rs` (in `game/`) - Server-side crafting logic and handlers

```rust src/game/crafting.rs
fn recipe_level_check_passed(player: &Player, recipe: &RecipeDefinition) -> bool {
    match recipe.category {
        RecipeCategory::Smithing => player.skills.smithing.level >= recipe.level_required,
        RecipeCategory::Alchemy => player.skills.alchemy.level >= recipe.level_required,
        RecipeCategory::Cooking | RecipeCategory::Fletching | RecipeCategory::Leatherworking => {

    }
```

## Recipe Definition Structure

Recipes are defined in TOML files located in `data/recipes/`. Each recipe specifies ingredients, results, requirements, and crafting parameters.

### Recipe Schema

<ParamField path="display_name" type="string" required>
  Human-readable name shown to players. Defaults to recipe ID with underscores replaced by spaces.
</ParamField>

<ParamField path="description" type="string">
  Descriptive text explaining what the recipe creates. Defaults to empty string.
</ParamField>

<ParamField path="category" type="enum" default="materials">
  Recipe category determining skill requirements:
  - `smithing` - Smithing skill
  - `alchemy` - Alchemy skill
  - `cooking`, `fletching`, `leatherworking` - Survivalist skill
  - `materials`, `consumables`, `equipment`, `tools` - Combat level
</ParamField>

<ParamField path="section" type="string">
  UI organization section within the category (e.g., "fish", "equipment", "restoration").
</ParamField>

<ParamField path="level_required" type="integer" default="0">
  Minimum skill level required to craft this recipe.
</ParamField>

<ParamField path="station" type="string">
  Required crafting station ID (e.g., "anvil", "furnace", "fire_pit"). If omitted, can craft anywhere.
</ParamField>

<ParamField path="craft_time_ms" type="integer" default="0">
  Crafting duration in milliseconds. 0 = instant crafting.
</ParamField>

<ParamField path="xp" type="integer" default="0">
  Experience points awarded on successful craft.
</ParamField>

<ParamField path="requires_discovery" type="boolean" default="false">
  If true, player must discover the recipe via recipe scroll before crafting.
</ParamField>

<ParamField path="required_tool" type="string">
  Tool item that must be in inventory (not consumed). Example: "hammer", "saw".
</ParamField>

<ParamField path="burn_result" type="string">
  Item ID produced when cooking burns (cooking only).
</ParamField>

<ParamField path="burn_stop_level" type="integer">
  Skill level at which burn chance reaches 0% (cooking only).
</ParamField>

<ParamField path="ingredients" type="array" required>
  Array of ingredient items with `item_id` and `count` (defaults to 1).
</ParamField>

<ParamField path="results" type="array" required>
  Array of result items with `item_id` and `count` (defaults to 1).
</ParamField>

## Timed Crafting

Recipes with `craft_time_ms > 0` use the timed crafting system. Materials are consumed immediately, and the player enters a crafting state until completion.

### Crafting State

```rust src/game.rs
pub struct CraftingState {
    pub recipe_id: String,
    pub started_at: std::time::Instant,
    pub duration_ms: u64,
    /// Materials consumed at start (for refund on cancel/interrupt)
    pub consumed_materials: Vec<(String, i32)>,
    /// Remaining crafts after this one completes (0 = single craft)
    pub batch_remaining: u32,
    /// Original total for batch progress display (1 = single craft)
    pub batch_total: u32,
}
```

### Timed Craft Flow

<Steps>
  <Step title="Validation">
    Server validates skill level, ingredients, inventory space, and station proximity.
  </Step>
  <Step title="Material Consumption">
    Materials are immediately removed from inventory (stored in `consumed_materials` for potential refund).
  </Step>
  <Step title="Crafting Started">
    Player enters crafting state. Server sends `CraftingStarted` message with duration.
  </Step>
  <Step title="Progress Tracking">
    Client displays progress bar. Server tracks elapsed time each tick.
  </Step>
  <Step title="Completion">
    When `started_at.elapsed() >= duration_ms`, server completes craft, awards items and XP.
  </Step>
  <Step title="Interruption Handling">
    If player dies or cancels, materials are refunded from `consumed_materials`.
  </Step>
</Steps>

### Completion Processing

```rust src/game/crafting.rs
pub async fn process_timed_crafting_completions(&self) {
    let mut players = self.players.write().await;
    
    for pid in player_ids {
        let should_complete = player.crafting_state.as_ref().is_some_and(|state| {
            state.started_at.elapsed().as_millis() as u64 >= state.duration_ms
        });
        
        if should_complete {
            let crafting = player.crafting_state.take().unwrap();
            let recipe = self.crafting_registry.get(&crafting.recipe_id).unwrap();
            
            // Check for burn (cooking only)
            let burned = check_burn(&recipe, player.skills.survivalist.level);
            
            // Award items and XP
            // Send completion messages
        }
    }
}
```

## Batch Crafting

Players can craft multiple items in sequence using the batch crafting system. Each item in the batch goes through a full timed craft cycle.

### Batch Craft Flow

<Steps>
  <Step title="Initiate Batch">
    Player requests batch craft with quantity. Server validates first craft and starts it.
  </Step>
  <Step title="Completion Check">
    When each craft completes, server checks `batch_remaining > 0`.
  </Step>
  <Step title="Continue or Stop">
    If materials and space available, consume next set and start next craft. Otherwise, batch ends.
  </Step>
  <Step title="Progress Updates">
    Server sends `CraftingBatchProgress` messages showing `completed/total`.
  </Step>
</Steps>

### Batch Continuation Logic

```rust src/game/crafting.rs
fn can_continue_batch(
    player: &Player,
    recipe: &RecipeDefinition,
    item_registry: &ItemRegistry,
) -> bool {
    let has_ingredients = recipe.ingredients.iter().all(|ingredient| {
        player.inventory.has_item(&ingredient.item_id, ingredient.count)
    });
    let has_space = recipe.results.iter().all(|result| {
        player.inventory.has_space_for(&result.item_id, result.count, item_registry)
    });
    
    has_ingredients && has_space
}
```

### "Craft All" Feature

Players can request `quantity = u32::MAX` to craft as many as possible:

```rust src/game/crafting.rs
let actual_quantity = if quantity == u32::MAX {
    let mut max_possible = u32::MAX;
    for ingredient in &recipe.ingredients {
        let have = player.inventory.count_item(&ingredient.item_id) as u32;
        let can_make = have / ingredient.count as u32;
        max_possible = max_possible.min(can_make);
    }
    max_possible.max(1)
} else {
    quantity.max(1)
};
```

## Cooking Burn Mechanics

Cooking recipes can fail and produce burnt items based on player skill level. Burn chance decreases linearly from 50% at `level_required` to 0% at `burn_stop_level`.

### Burn Calculation

```rust src/game.rs
/// Check if a cooking recipe burns based on player level.
/// Returns true if the food should burn.
/// Burn chance: 50% at recipe's level_required, linearly decreasing to 0% at burn_stop_level.
fn check_burn(recipe: &crate::crafting::definition::RecipeDefinition, player_level: i32) -> bool {
    if let (Some(_burn_result), Some(burn_stop)) = (&recipe.burn_result, recipe.burn_stop_level) {
        if player_level >= burn_stop {
            return false; // Never burns at or above stop level
        }
        let level_range = (burn_stop - recipe.level_required) as f64;
        if level_range <= 0.0 {
            return false;
        }
        let burn_chance = ((burn_stop - player_level) as f64 / level_range) * 0.5;
        rand::random::<f64>() < burn_chance
    } else {
        false // No burn mechanic for this recipe
    }
}
```

### Burn Example

For `cook_shrimp` with `level_required = 1` and `burn_stop_level = 5`:

| Player Level | Burn Chance |
|--------------|-------------|
| 1            | 50%         |
| 2            | 37.5%       |
| 3            | 25%         |
| 4            | 12.5%       |
| 5+           | 0%          |

### Burnt Item Rewards

When food burns, the player receives the `burn_result` item instead of normal results, and earns half XP:

```rust src/game/crafting.rs
let burned = check_burn(&recipe, player.skills.survivalist.level);
let mut items_gained = Vec::new();

if burned {
    let burn_item = recipe.burn_result.as_ref().unwrap();
    player.inventory.add_item(burn_item, 1, &self.item_registry);
    items_gained.push((burn_item.clone(), 1));
} else {
    for result in &recipe.results {
        player.inventory.add_item(&result.item_id, result.count, &self.item_registry);
        items_gained.push((result.item_id.clone(), result.count as u32));
    }
}

let xp_gained = if burned { recipe.xp / 2 } else { recipe.xp };
```

## Recipe Discovery

Recipes with `requires_discovery = true` must be unlocked before crafting. Players learn recipes by using recipe scroll items.

### Discovery Flow

<Steps>
  <Step title="Obtain Recipe Scroll">
    Player acquires a recipe scroll item (e.g., `recipe_spine_blade` from loot or quests).
  </Step>
  <Step title="Use Scroll">
    Player uses the scroll from inventory. Server validates and consumes it.
  </Step>
  <Step title="Recipe Unlocked">
    Recipe ID is added to `player.discovered_recipes` HashSet and persisted to database.
  </Step>
  <Step title="Notification">
    Client receives `RecipeDiscovered` message and displays the unlocked recipe.
  </Step>
</Steps>

### Discovery Handler

```rust src/game/crafting.rs
pub async fn handle_use_recipe_scroll(&self, player_id: &str, slot_index: u8) {
    let mut players = self.players.write().await;
    let player = players.get_mut(player_id).unwrap();
    
    let item_id = player.inventory.slots[slot_index].item_id.clone();
    let recipe_id = item_id.strip_prefix("recipe_").unwrap();
    
    // Validate recipe exists
    if self.crafting_registry.get(&recipe_id).is_none() {
        self.send_system_message(player_id, "This recipe scroll is for an unknown recipe.").await;
        return;
    }
    
    // Check if already known
    if player.discovered_recipes.contains(&recipe_id) {
        self.send_system_message(player_id, "You already know this recipe.").await;
        return;
    }
    
    // Consume scroll
    player.inventory.slots[slot_index].quantity -= 1;
    if player.inventory.slots[slot_index].quantity <= 0 {
        player.inventory.slots[slot_index] = None;
    }
    
    // Unlock recipe
    player.discovered_recipes.insert(recipe_id.clone());
    
    // Persist to database
    if let Some(ref db) = self.db {
        db.save_discovered_recipe(character_id, &recipe_id).await;
    }
    
    // Notify client
    self.send_to_player(player_id, ServerMessage::RecipeDiscovered { recipe_id }).await;
}
```

## Experience and Skill Progression

Crafting awards experience to the appropriate skill based on recipe category:

### XP Distribution

```rust src/game/crafting.rs
fn add_crafting_xp(
    player: &mut Player,
    category: RecipeCategory,
    xp_gained: u32,
) -> Vec<CraftXpUpdate> {
    let mut xp_results = Vec::new();
    if xp_gained == 0 {
        return xp_results;
    }
    
    let xp_gained = xp_gained as i64;
    
    match category {
        RecipeCategory::Smithing => {
            let leveled = player.skills.smithing.add_xp(xp_gained);
            xp_results.push((SkillType::Smithing, xp_gained, 
                player.skills.smithing.xp, player.skills.smithing.level, leveled));
        }
        RecipeCategory::Alchemy => {
            let leveled = player.skills.alchemy.add_xp(xp_gained);
            xp_results.push((SkillType::Alchemy, xp_gained,
                player.skills.alchemy.xp, player.skills.alchemy.level, leveled));
        }
        RecipeCategory::Cooking | RecipeCategory::Fletching | RecipeCategory::Leatherworking => {
            let leveled = player.skills.survivalist.add_xp(xp_gained);
            xp_results.push((SkillType::Survivalist, xp_gained,
                player.skills.survivalist.xp, player.skills.survivalist.level, leveled));
        }
        _ => {} // Materials, consumables, equipment, tools award no XP
    }
    
    xp_results
}
```

## Registry Loading

The `CraftingRegistry` loads all recipe TOML files from `data/recipes/` at server startup:

```rust src/crafting/registry.rs
impl CraftingRegistry {
    pub fn load_from_directory(&mut self, data_dir: &Path) -> Result<(), String> {
        let recipes_dir = data_dir.join("recipes");
        
        let entries = std::fs::read_dir(&recipes_dir)?;
        
        for entry in entries {
            let path = entry.path();
            
            if path.extension().map_or(false, |ext| ext == "toml") {
                let content = std::fs::read_to_string(&path)?;
                
                // Parse as table of recipes
                let table: HashMap<String, RawRecipeDefinition> = toml::from_str(&content)?;
                
                for (id, raw) in table {
                    let recipe = RecipeDefinition::from_raw(&id, &raw);
                    self.recipes.insert(id, recipe);
                }
            }
        }
        
        info!("Loaded {} recipe definitions", self.recipes.len());
        Ok(())
    }
    
    pub fn get(&self, id: &str) -> Option<&RecipeDefinition>
    pub fn all(&self) -> impl Iterator<Item = &RecipeDefinition>
    pub fn by_category(&self, category: RecipeCategory) -> Vec<&RecipeDefinition>
    pub fn to_client_definitions(&self) -> ServerMessage
}
```

## Crafting Validation

The server performs comprehensive validation before allowing crafts:

<Steps>
  <Step title="Recipe Exists">
    Verify recipe ID is registered in `CraftingRegistry`.
  </Step>
  <Step title="Player Active">
    Confirm player is connected, alive, and not already crafting.
  </Step>
  <Step title="Skill Level">
    Check player skill level meets `level_required` for recipe category.
  </Step>
  <Step title="Station Access">
    If recipe requires station, verify player is within `interaction_range` tiles.
  </Step>
  <Step title="Tool Requirement">
    If `required_tool` specified, verify tool exists in inventory (not consumed).
  </Step>
  <Step title="Recipe Discovery">
    If `requires_discovery = true`, verify recipe in `discovered_recipes` set.
  </Step>
  <Step title="Ingredients">
    Verify all ingredients present in sufficient quantities.
  </Step>
  <Step title="Inventory Space">
    Verify sufficient empty slots for all result items (considering stackability).
  </Step>
</Steps>

## Advanced Features

### Material Refunding

When timed crafts are cancelled or interrupted (player death), consumed materials are refunded:

```rust src/game/crafting.rs
pub async fn cancel_crafting(&self, player_id: &str, reason: &str) {
    let mut players = self.players.write().await;
    let player = players.get_mut(player_id).unwrap();
    
    if let Some(crafting) = player.crafting_state.take() {
        // Refund consumed materials
        for (item_id, count) in &crafting.consumed_materials {
            player.inventory.add_item(item_id, *count, &self.item_registry);
        }
        
        self.send_to_player(player_id, ServerMessage::CraftingCancelled {
            reason: reason.to_string(),
        }).await;
    }
}
```

```rust src/crafting/registry.rs
pub fn to_client_definitions(&self) -> ServerMessage {

                item_name: i.item_id.clone(), // Client resolves display name

            }).collect(),
            results: recipe.results.iter().map(|r| RecipeResult {
                item_id: r.item_id.clone(),
                item_name: r.item_id.clone(), // Client resolves display name
                count: r.count,
            }).collect(),
            station: recipe.station.clone(),
            craft_time_ms: recipe.craft_time_ms,
            xp: recipe.xp,

            burn_result: recipe.burn_result.clone(),
```

## Performance Considerations

<Warning>
The timed crafting completion check runs every server tick (20Hz). To minimize overhead:
- Use `is_some_and()` to short-circuit when no crafting state exists
- Only calculate elapsed time for active crafting players
- Batch all completion processing in a single lock acquisition
</Warning>

```rust src/game/crafting.rs
pub async fn process_timed_crafting_completions(&self) {

            if !should_complete {

        }
        
        completions

}
```
